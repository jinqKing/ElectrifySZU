## Slide 4 — 架构 Architecture

**Eyebrow：** Architecture

**标题：** 静态页面负责体验，本地后端代理校园网查询

**副文：** GitHub Pages 部署公开演示页；真实查询时运行本地服务，后端代理访问校园电费系统，前端只消费统一 JSON API。

**流程图：**

| Step | 名称 | 描述 |
|------|------|------|
| 01 | **输入宿舍信息** | 校区、楼栋（模糊搜索+自动补全）、房间号、统计天数 |
| 02 | **发现 roomId** | 把可读房间号转换为电费系统内部参数 |
| 03 | **API 代理查询** | 由校园网环境中的 Python 服务请求内网接口 |
| 04 | **仪表盘渲染** | 余额、趋势、充值记录、预警状态、预计可用天数同步更新 |

**技术栈标注：** Python (uv) + 原生 HTML/CSS/JS + CSV 存储 + SMTP 邮件

## Slide 6 — 订阅预警系统（核心差异）Subscription

**Eyebrow：** Killer Feature

**标题：** 从"想起来了才查"变成"系统替你照看"

**副文：** 这是 ElectrifySZU 最区别于其他查询工具的能力——不是一次性查询，而是持续守护。

**订阅流程：**

| 环节 | 说明 |
|------|------|
| 📧 **邮箱自动补全** | 输入学号自动补全后缀，仅支持深大邮箱 |
| 🔔 **双类型订阅** | 低电量预警 + **每日电费报告**（独立开关，可单选/全选） |
| ✉️ **验证确认** | 提交后发送验证邮件 → 点击链接激活 → 弹窗通知结果 → badge 显示"待验证/已订阅" |
| 🔄 **订阅管理** | 仪表盘内 toggle 开关，一键退订（token 一次性，用后即销毁） |
| ⏰ **每日限一次** | 低于阈值时每天最多发一封预警邮件，不骚扰用户 |
| 🧪 **测试模式** | `FORCE_SEND_ALERT` 环境变量可绕过电费 API 直接触发预警，方便演示 |

**信号指示器（视觉）：**

| 级别 | 阈值 |
|------|------|
| 电量充足 | > 阈值 |
| 低电量 | ≤ 阈值 |
| 即将耗尽 | ≤ 阈值 × 0.5 |

## Slide 7 — 工程品质与开发历程 Engineering

**Eyebrow：** Engineering

**标题：** 68 次提交，19 个测试用例，3 天高强度迭代

**副文：** 两个开发者，一个负责全功能测试，一个负责服务器部署上线。细粒度任务分配，不断打磨。

**开发时间线：**

| 时间 | 里程碑 | 产出 |
|------|--------|------|
| **24h** | 基础功能跑通 | 前端原型验证：查询面板 + 指标卡 + 趋势图 |
| **48h** | 服务器稳定 | 大量 branch/worktree 并行尝试优化；订阅验证流程上线 |
| **72h** | 加强安全 + 开放测试 | 邮件重试机制、graceful shutdown、统一错误码、结构化日志 |

**工程实践亮点：**

| 实践 | 说明 |
|------|------|
| 🧪 **19 例 pytest** | 覆盖 demo 数据生成、楼栋合并去重、store CRUD |
| 📊 **结构化日志** | 不同模块日志用不同颜色区分；请求含 IP + 方法 + 路径 + 耗时 |
| 🛡️ **Graceful Shutdown** | 收到终止信号后 drain 进行中的请求再关闭 socket |
| 🏥 **Health/Version 端点** | `/api/health` + `/api/version` 供运维监控 |
| 🔁 **邮件重试** | SMTP 发送失败自动重试，异常与业务错误分离 |
| 🎨 **一个按钮改三版** | 订阅按钮从设计到交互经历了三轮迭代 |

**原则：** 减少工具依赖（先用 CSV 跑起来）、细粒度任务、持续打磨。
